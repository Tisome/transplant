# 《基于FPGA的超声波时差测量模块》论文架构参考

## 一、先统一全文主线

这篇论文的主角应始终是“超声波时差测量模块”，而不是完整产品功能堆砌。全文最好围绕下面这条主线展开：

1. 为什么超声波时差测量有意义。
2. 时差法测量的关键难点是什么。
3. 为什么要用 FPGA 做前级高速处理。
4. FPGA 如何提取与时间差相关的关键特征。
5. MCU 如何完成 `dt` 解算并进一步输出流量。
6. 在缺少完整装管实测条件下，如何通过仿真和板级联调证明方案可行。

对应到论文表达上，建议始终坚持一句话：

“本文研究的是基于 FPGA 的超声波时差测量模块设计与实现，流量输出是该测量模块在系统层面的后续应用结果。”

## 二、推荐的章节结构

比较适合你的，是 `7` 章结构。

### 第 1 章 绪论

这一章回答三个问题：

1. 为什么要做超声波时差测量。
2. 当前研究主要卡在哪些环节。
3. 本文准备解决什么问题。

建议内容如下：

1. `1.1 研究背景与研究意义`
2. `1.2 国内外研究现状`
3. `1.3 本文主要研究内容`
4. `1.4 论文组织结构`

这一章的作用是把题目立住，不要过早陷入具体代码和工程细节。

### 第 2 章 超声波时差测量原理与系统需求分析

这一章是理论基础章，重点不是讲产品，而是讲“为什么这样测”。

建议内容如下：

1. `2.1 时差法超声波测量原理`
2. `2.2 顺流与逆流传播时间关系`
3. `2.3 时间差与流速、流量的换算关系`
4. `2.4 时间差测量中的主要误差来源`
5. `2.5 系统对时间分辨率、实时性和抗干扰能力的要求`

这一章的重点要落在：

1. `Δt` 很小，测起来难。
2. 后面所有流量计算都建立在 `Δt` 测得准的前提下。

### 第 3 章 系统总体方案与数据流程设计

这一章回答的是：

“整套系统怎样围绕时差测量模块组织起来？”

建议内容如下：

1. `3.1 系统总体架构`
2. `3.2 FPGA 与 MCU 的任务划分`
3. `3.3 信号流与数据流设计`
4. `3.4 特征包格式设计`
5. `3.5 验证平台与应用平台说明`

这一章建议明确写出数据主链路：

`超声采样信号 -> FPGA 相关计算与峰值特征提取 -> 特征包上传 -> MCU 解算 dt -> 流量输出`

这里要强调的是“特征上传”而不是“整段波形上传”，因为这是你方案的重要合理性来源。

### 第 4 章 基于 FPGA 的超声波时差测量模块设计

这一章是全文核心。

建议内容如下：

1. `4.1 模块设计目标`
2. `4.2 前级激励与采样时序设计`
3. `4.3 采样数据缓存与处理流程`
4. `4.4 互相关/卷积处理模块设计`
5. `4.5 峰值搜索与特征提取模块设计`
6. `4.6 特征打包与接口输出设计`
7. `4.7 本章小结`

这一章重点要解释清楚三件事：

1. 为什么相关计算放在 FPGA 中完成。
2. 为什么只输出 `idxA/idxB/y1/y2/y3` 这类关键特征，而不是直接输出整段波形。
3. FPGA 设计如何服务于后级时间差精细解算。

如果你本地已有 FPGA 文档里包含 `卷积模块`、`timeget 模块`、`通信模块`，这一章可以直接吸收进去。

### 第 5 章 MCU 端时间差解算与流量输出实现

这一章虽然写 MCU，但主角仍然是“时间差解算”。

建议内容如下：

1. `5.1 特征包接收与解析流程`
2. `5.2 峰值邻域数据的抛物线插值方法`
3. `5.3 传播时间差 dt 的解算过程`
4. `5.4 由时间差到流量的换算方法`
5. `5.5 异常剔除与结果平滑处理`
6. `5.6 瞬时流量与累计流量输出`
7. `5.7 工程化辅助功能简述`

这一章要把重点放在：

1. `idxA/idxB/y1/y2/y3 -> t1/t2/dt`
2. `dt -> flow`
3. 结果为什么还需要滤波、异常处理和稳定化输出

`LVGL`、按键去抖、`Modbus` 这些内容最多只放在 `5.7` 作为工程辅助功能简述，点到为止，不要抢主线。

### 第 6 章 系统测试与验证方法

这一章非常关键，因为它回答：

“在没有完整管道实测条件下，如何证明你的模块是成立的？”

建议内容如下：

1. `6.1 验证目标与总体思路`
2. `6.2 MATLAB 仿真平台构建`
3. `6.3 时差提取正确性验证`
4. `6.4 流量输出正确性验证`
5. `6.5 噪声条件下的鲁棒性分析`
6. `6.6 统计评价指标与结果表达方式`
7. `6.7 板级联调验证`
8. `6.8 本章小结`

这一章建议按三层验证逻辑来写：

1. 先验证 `dt` 算得对不对。
2. 再验证基于 `dt` 的流量输出是否合理。
3. 最后验证 FPGA 到 MCU 的链路是否真的跑通。

也就是说，验证重点应先落在“时差测量模块”，再落到“流量输出结果”。

为了让第六章更有说服力，建议把“加噪声仿真 + 统计指标评价”明确写进去，而不只是展示几张波形图或误差曲线。

### 第 6 章中建议补充的验证指标

在 `6.3` 到 `6.6` 中，建议不仅给出单次误差，还对多次仿真结果进行统计分析。比较适合写进论文的指标包括：

1. `均值偏差（Bias）`
2. `标准差（STD）`
3. `均方根误差（RMSE）`
4. `95% 置信区间`
5. `离群率`

这些指标分别反映不同问题：

1. `Bias` 反映算法是否存在系统性偏差，也就是“算出来是否总偏大或偏小”。
2. `STD` 反映结果波动程度，也就是“稳定不稳定”。
3. `RMSE` 同时考虑偏差和波动，适合用来评价整体误差水平。
4. `95% 置信区间` 反映在重复实验条件下结果分布范围，能体现估计结果的可靠程度。
5. `离群率` 用于评价算法在噪声干扰下出现明显失准结果的频率，适合体现鲁棒性。

### 第 6 章更具体的写法建议

#### `6.3 时差提取正确性验证`

这一节先做无噪声验证，再做低噪声验证，重点看 `dt_true` 与 `dt_est` 的差异。

建议展示：

1. `dt` 真值与估计值对比图
2. `dt` 绝对误差曲线
3. 不同流速点下的 `Bias`、`STD`、`RMSE`

这一节主要证明：

“所设计的时差测量模块在理想和低干扰条件下能够正确恢复传播时间差。”

#### `6.4 流量输出正确性验证`

这一节把重点放在 `Q_true` 与 `Q_est` 的比较，而不是只停留在 `dt`。

建议展示：

1. 流量真值与估计值对比图
2. 瞬时流量相对误差
3. 累计流量误差
4. 不同工况下的 `RMSE`

这一节主要证明：

“基于所解算的时间差，系统能够进一步得到合理的流量输出结果。”

#### `6.5 噪声条件下的鲁棒性分析`

这一节建议专门写“加噪声仿真”，这是你论文里很值得展开的一块。

可采用的写法是：

1. 在 MATLAB 中构建含噪超声信号或含噪特征模型。
2. 分别设置多个 `SNR` 条件。
3. 在每个噪声等级下重复多次仿真。
4. 统计 `dt` 和流量输出误差。

建议展示：

1. `SNR-Bias` 曲线
2. `SNR-STD` 曲线
3. `SNR-RMSE` 曲线
4. `SNR-离群率` 曲线

如果你愿意做得更完整，还可以在这一节写“Monte Carlo 重复试验”，也就是在同一参数条件下多次随机生成噪声，统计分布结果。这样这些统计指标就有了明确来源，而不是只算一次结果。

#### `6.6 统计评价指标与结果表达方式`

这一节可以专门说明你在论文中如何定义这些指标，避免老师答辩时问“这些量是怎么来的”。

建议内容如下：

1. `Bias`
   可定义为多次实验中估计值与真值误差的平均值。
2. `STD`
   可定义为误差序列的标准差。
3. `RMSE`
   可定义为误差平方均值开根号，用于评价整体误差水平。
4. `95% 置信区间`
   可用于描述在重复实验下误差分布的大致范围。
5. `离群率`
   可定义为误差超过某一阈值的样本数占总样本数的比例。

这一节还可以明确说明：

1. `Bias` 偏小，说明系统性偏差小。
2. `STD` 偏小，说明输出稳定。
3. `RMSE` 偏小，说明综合误差低。
4. `95% 置信区间` 越窄，说明结果越集中。
5. `离群率` 越低，说明抗干扰和鲁棒性越好。

#### `6.7 板级联调验证`

这一节重点不再是统计精度，而是证明链路真的跑通。

建议写：

1. FPGA 是否稳定输出特征包
2. MCU 是否正确接收并解析
3. `dt` 与流量结果是否能实时刷新
4. 多次联调下系统是否运行稳定

这一节可以少放复杂公式，多放流程图、串口日志、调试结果截图或关键运行数据。

### 第 6 章最终想呈现的效果

如果第六章按上面方式组织，那么它就不是简单的“做了几个实验”，而是形成下面这套逻辑：

1. 用无噪声验证证明算法正确。
2. 用含噪声验证证明算法有鲁棒性。
3. 用 `Bias`、`STD`、`RMSE`、`95%置信区间`、`离群率` 量化误差特征。
4. 用板级联调证明系统可实现。

这样写出来，第六章会明显更像一章完整的“验证章节”，也更接近正式毕业论文的表达方式。

### 第 7 章 总结与展望

建议内容如下：

1. 总结本文完成的设计与验证工作。
2. 总结该时差测量模块的主要特点和效果。
3. 说明当前工作的不足。
4. 给出后续改进方向。

不足建议诚实写：

1. 尚未完成标准流量装置下的完整实流标定。
2. 尚未完成长时间装管稳定性测试。
3. 温漂、复杂流态补偿等内容仍可继续完善。

## 三、每章的轻重分配建议

从整篇论文的篇幅分配看，建议大致按下面的重心来写：

1. 第 1 章：中等篇幅，用于立题和交代研究定位。
2. 第 2 章：中等篇幅，用于讲清测量原理与误差来源。
3. 第 3 章：中等篇幅，用于交代系统架构和数据流程。
4. 第 4 章：最大篇幅，作为全文核心。
5. 第 5 章：较大篇幅，重点写 `dt` 解算与流量输出。
6. 第 6 章：较大篇幅，重点写测试与验证。
7. 第 7 章：较小篇幅，简洁总结与展望。

如果按内容重要性排序，建议优先保证下面三章写扎实：

1. 第 4 章 FPGA 模块设计
2. 第 5 章 MCU 端时间差解算
3. 第 6 章 测试与验证

## 四、哪些内容应重点写，哪些内容应轻写

### 应重点写的

1. 超声波时差测量原理
2. 时间差为什么难测
3. 互相关/峰值搜索为什么有效
4. FPGA 为什么适合做前级高速处理
5. 特征包设计为什么合理
6. 抛物线插值为什么能提高时间差解算精度
7. 仿真和板级联调如何支撑论文结论

### 应次重点写的

1. 由 `dt` 到流量的换算关系
2. 数据平滑、异常剔除与稳定化输出
3. `GD32F303RCT6` 上的联调实现
4. 向后续平台移植的说明

### 只需轻写的

1. `LVGL` 显示
2. 按键去抖
3. `Modbus` 通信
4. 菜单界面
5. 一般性工程配套代码

这些内容可以写，但最好只作为“工程化辅助功能”简要介绍，不能写成论文主线。

## 五、这一架构最适合你的原因

这套结构最适合你的原因在于：

1. 它和论文题目完全一致，核心始终是“超声波时差测量模块”。
2. 它能自然容纳你现有代码结构，即 FPGA 前处理、MCU 后级解算、最终输出流量。
3. 它能解释为什么即使当前没有完整装管实测，论文仍然可以通过仿真与联调完成较完整的验证。
4. 它避免把论文重心错误地放到显示界面、通信协议或外围工程细节上。

## 六、一句话版总结构思

如果把整篇论文压缩成一句话，可以理解为：

“本文围绕超声波时差测量这一核心问题，设计了基于 FPGA 的前级特征提取模块，构建了 MCU 后级时间差解算与流量输出链路，并通过 MATLAB 仿真与板级联调验证了该方案的正确性与可实现性。”
