# 第四章与第五章撰写提示

这份文档不是论文正文，而是给后续写作时快速参考的提示稿。  
目标是帮助在有限时间内把第四章和第五章写得更聚焦，避免写散、写重复，或把大量篇幅耗在不重要的工程细节上。

---

## 一、先把两章的分工想清楚

第四章和第五章最容易出现的问题，是两章都在重复讲同一条处理链。为了避免这种情况，可以先固定分工：

- 第四章回答：`FPGA 前端是怎样完成时差相关特征提取的？`
- 第五章回答：`MCU 后端是怎样基于这些特征完成 dt 解算和流量输出的？`

也就是说：

- 第四章的重点是“前端模块设计”
- 第五章的重点是“后端算法与输出实现”

一句话区分：

> 第四章主要写“如何得到关键特征”，第五章主要写“如何利用关键特征得到时间差和流量结果”。

---

## 二、第四章写什么

推荐第四章名称：

`第4章 基于FPGA的超声波时差测量模块设计`

这一章是全文核心，应该是整篇论文里技术密度最高的一章。

### 4.1 设计目标与总体思路

这一节建议写三件事：

1. 本模块在系统中的定位  
   说明 FPGA 模块负责超声发射控制、采样同步、前级相关处理和关键特征提取。

2. 本模块要解决的问题  
   说明原始采样数据量大、时序要求高、时间差很小，因此需要在前端完成高速处理。

3. 本模块的总体思路  
   说明系统采用“FPGA 前端提取特征，MCU 后端精细解算”的结构。

这一节不要马上掉进每个子模块的细节。  
它更像第四章的“开场”，作用是先把整章思路立住。

可直接围绕这几个关键词写：

- 高速
- 并行
- 确定性时序
- 特征提取
- 减少跨平台数据传输

### 4.2 激励与采样时序设计

这一节可以重点写：

1. FPGA 负责产生换能器激励信号  
   不是换能器自己发射，而是 FPGA 内部 `pwm` 模块控制发射节拍。

2. 发射与采样之间的关系  
   发射结束后由 FPGA 触发后续采样，保证采样窗口与回波到达相匹配。

3. 测量周期组织方式  
   当前系统是两个 `4 ms` 子周期组成一组完整测量，约 `8 ms` 形成一组 A/B 特征结果。

这一节适合放：

- 一个简化流程图
- 一个单周期时序图

写法建议：

- 重点说明“先发射、再采样、再处理”的因果关系
- 不必过度展开底层 HDL 语句
- 不必把所有时钟域同步细节写得太碎

### 4.3 采样数据缓存与处理流程

这一节回答的是：

`采样数据进入 FPGA 后，如何被缓存并送入后续处理模块？`

可以写：

1. ADC 采样数据先进入内部缓存
2. 缓存后的数据再按算法处理节拍读出
3. 前端数据流在 FPGA 内部完成闭环处理，而不是直接外发

这一节不要和 `3.3 数据流设计` 重复太多。  
第三章讲的是系统级流程，这一节讲的是 FPGA 内部处理路径。

比较适合的写法是：

- 第三章：系统层面的“数据怎么流”
- 第四章：模块层面的“数据在 FPGA 里怎么处理”

### 4.4 相关/卷积处理模块设计

这一节很关键，是第四章的技术主干之一。

建议写：

1. 为什么采用相关或卷积处理  
   说明其作用是增强回波特征、突出峰值位置。

2. 模块输入输出是什么  
   输入为采样数据和参考模板，输出为相关或卷积结果序列。

3. 模块处理流程是什么  
   例如滑动窗口、乘加累积、有效输出等。

4. 为什么适合在 FPGA 中实现  
   强调并行计算和时序确定性。

这一节不要把数学理论讲太多，第二章已经讲过原理。  
这里更适合强调：

- 模块怎么工作
- 为什么这样设计
- 这样设计带来了什么好处

如果你要写得更像论文，可以按：

`设计目的 -> 输入输出 -> 核心结构 -> 工作流程 -> 设计意义`

### 4.5 峰值搜索与特征提取模块设计

这一节要和 `4.4` 区分开。

`4.4` 讲的是“如何得到相关结果序列”  
`4.5` 讲的是“如何从相关结果序列中提取关键特征”

建议重点写：

1. 为什么只保留关键特征而不保留整段波形
2. 峰值搜索模块如何找到最大值及其索引
3. 为什么还要保留峰值邻域信息
4. 这些特征如何服务于 MCU 侧插值解算

这里特别适合把 `idxA / idxB / y1 / y2 / y3` 的设计意义讲清楚。

推荐的论述逻辑：

1. 峰值索引用于给出大致位置
2. 邻域三点用于峰值细化
3. 两者结合，既压缩数据量，又保留了解算所需信息

### 4.6 特征打包与接口输出设计

这一节主要写接口，不要再重讲前面已经讲过的卷积和峰值搜索。

适合写：

1. 数据包字段组成
2. 数据包长度为什么是 `22 Byte`
3. 为什么要等一组完整特征齐备后再发送
4. SPI 从机方式的组织逻辑

这节可以直接和第三章 `3.4` 呼应，但第四章这里更偏实现层。

换句话说：

- `3.4` 讲“为什么这样设计包”
- `4.6` 讲“包在 FPGA 侧怎样形成并输出”

### 4.7 本章小结

这节不用长。  
100 到 200 字左右就够。

建议只总结：

1. 本章完成了 FPGA 前端模块设计
2. 说明了发射、采样、相关处理、峰值提取和特征输出的实现思路
3. 为后续 MCU 端时间差解算提供了特征输入基础

## 三、第四章写作提醒

第四章最容易出现的问题有三个：

### 1. 过度像源码说明书

不要每节都写成：

- 某模块有几个输入
- 某模块有几个输出
- 某寄存器何时置位

这些可以点到为止，但不能让整章变成 Verilog 阅读笔记。

更好的写法是：

- 先说模块目的
- 再说工作机制
- 最后说这样设计的意义

### 2. 和第三章重复

如果第三章已经说过“系统数据从 FPGA 到 MCU”，第四章不要再用整段篇幅重新描述系统总体流程。  
第四章要下沉到模块设计本身。

### 3. 和第五章抢内容

第四章不要细讲：

- 抛物线插值公式
- `dt` 解算细节
- 流量换算
- 结果平滑

这些都应该放到第五章。

## 四、第五章写什么

推荐第五章名称：

`第5章 MCU端时间差解算与流量输出实现`

这一章虽然写 MCU，但核心仍然不是“单片机程序”，而是：

`如何根据 FPGA 给出的关键特征，得到 dt 和流量结果`

### 5.1 特征包接收与解析流程

这一节写得简洁一点就好。

重点写：

1. MCU 通过 SPI 接收 FPGA 特征包
2. 特征包中包含 `idxA / idxB / y1 / y2 / y3`
3. 解析后形成后级解算的输入

这里不要把 SPI 细节写太重，也不要写成驱动代码说明书。

这一节的目标只是回答：

`后端算法的输入从哪里来、是什么格式`

### 5.2 峰值邻域数据的抛物线插值方法

这是第五章最重要的小节之一。

建议写：

1. 为什么峰值索引只能得到整数采样点位置
2. 为什么要利用邻域三点做插值
3. 抛物线插值的基本公式
4. 插值结果如何提高时间分辨率

这一节可以适当有公式。  
但公式写完之后，一定要配一句解释，不然容易显得生硬。

比较好的写法是：

- 先讲问题：整数索引精度不够
- 再讲办法：利用三点局部拟合
- 最后讲效果：得到亚采样级峰值位置

### 5.3 传播时间差 `dt` 的解算过程

这一节要把后端核心过程写清楚：

`idxA / idxB / y1 / y2 / y3 -> t1 / t2 / dt`

建议写：

1. 先由峰值索引确定大致位置
2. 再由插值结果进行细化
3. 分别得到顺流与逆流传播时间
4. 最终得到 `dt`

这一节建议尽量把符号统一：

- `t1`、`t2`
- `dt`
- `Ts`

不要一会儿 `delta_t`，一会儿 `tof`，一会儿又写别的。

### 5.4 由时间差到流速、流量的换算方法

这一节承接第二章原理，但要落到实现。

建议写：

1. 根据 `dt` 求流速
2. 根据管道截面积求瞬时流量
3. 进一步得到累计流量

这里要注意一件事：

你的论文主线仍然是“时差测量模块”，所以这节要写，但不要比 `5.2`、`5.3` 更重。

写法上建议突出：

- `dt` 是核心中间量
- 流量输出是系统层面的后续应用结果

### 5.5 异常剔除与结果平滑处理

这一节建议写，但不要写太厚。

适合写：

1. 为什么原始结果会波动
2. 为什么要做异常剔除
3. 为什么要做结果平滑
4. 当前系统采用了哪些策略

这节的功能是增强工程合理性，让老师知道你不是“算一次就直接显示”。

但要注意：

这一节是辅助性的，不能写得比 `dt` 解算本身还重。

### 5.6 瞬时流量与累计流量输出

这节可以比较简洁。

重点写：

1. 系统最终输出哪些量
2. 瞬时流量与累计流量分别有什么意义
3. 输出结果如何供显示或通信使用

如果你愿意，这里也可以轻提一句：

> 系统具备基本的人机交互与数据显示能力，用于结果观察与工程调试。

### 5.7 工程化辅助功能简述

这一节一定要轻写。

这里最多放：

- 显示
- 参数查看
- 通信
- 简单交互

不要展开写：

- LVGL 页面细节
- 按键去抖算法细节
- Modbus 寄存器表大段说明

这一节的定位只能是：

`工程辅助功能，不是本文研究重点`

## 五、第五章写作提醒

第五章最容易出的问题有四个：

### 1. 把 MCU 写成“普通软件实现说明”

不要把大量篇幅放在：

- 任务创建
- 队列收发
- 定时器初始化
- 菜单显示逻辑

这些都不是第五章的重点。

### 2. 忘了主角还是 `dt`

即便第五章最后会写到流量输出，也不要忘记：

第五章真正最重要的是：

- 插值
- 时间差细化
- 时间差解算

### 3. 结果处理写太散

滤波、异常剔除、平滑、累计量这些内容最好集中到一两节里写，不要每个小节都提一遍。

### 4. 和第六章重复

第五章是“怎么实现”，第六章是“效果怎么样”。  
第五章不要提前放太多实验图和统计结果。

## 六、你写第四章和第五章时可以反复自问的几个问题

### 写第四章时问自己

1. 这一节讲的是哪个 FPGA 模块？
2. 这个模块为什么需要存在？
3. 它解决了什么问题？
4. 它的输入和输出是什么？
5. 它如何服务于后续时间差解算？

### 写第五章时问自己

1. 这一节是在解释哪一步后级解算？
2. 它和 `dt` 的关系是什么？
3. 它是主流程，还是辅助处理？
4. 这段内容会不会应该放到第六章验证里？

---

## 七、如果时间紧，第四章和第五章优先保证哪些部分

如果时间很紧，建议优先保证下面这些小节写扎实：

### 第四章优先级最高的

1. `4.1 设计目标与总体思路`
2. `4.4 相关/卷积处理模块设计`
3. `4.5 峰值搜索与特征提取模块设计`
4. `4.6 特征打包与接口输出设计`

### 第五章优先级最高的

1. `5.2 抛物线插值方法`
2. `5.3 dt 解算过程`
3. `5.4 时间差到流量的换算`
4. `5.5 异常剔除与结果平滑`

换句话说，即使其他部分写得简一点，也尽量保证：

`FPGA 如何提特征` 和 `MCU 如何算 dt`

这两部分最扎实。

---

## 八、最后给你的一个最实用建议

你现在自己写第四章和第五章时，可以始终抓住下面这条主线，不容易跑偏：

> FPGA 负责把原始超声采样信号压缩成与时间差相关的关键特征，MCU 负责基于这些特征完成时间差细化解算，并进一步得到流量输出结果。

只要每一节都能和这句话挂上钩，基本就不会写散。
