# 第一章 实习基本情况

## 1.1 实习项目与实习单位概况

### 1.1.1 实习项目简介

本人参加的实习项目为管道流体测量系统中 MCU 数据处理与通信模块的设计和验证。该项目面向超声波时差式流量测量场景，以 STM32F103 微控制器作为核心控制单元，围绕测量数据接收、特征数据解析、传播时间差计算、流速流量换算、结果平滑、参数管理以及通信输出等功能展开。实习期间，本人主要完成 STM32 平台上的数据处理与通信模块设计，使系统能够接收固定格式的测量特征数据包，并将其转换为便于上位机读取和现场使用的流体参数。

从系统应用场景看，管道流体测量设备需要长期安装在工业或实验环境中，对数据接收的实时性、计算结果的稳定性和外部通信接口的可用性都有一定要求。测量采集单元会向 STM32 输出包含索引值和峰值邻域信息的特征数据包，MCU 需要在较短时间内完成数据接收、字段解析和流体参数计算。与仅完成单个外设驱动不同，本次实习工作更接近一个嵌入式测量终端的软件设计，需要同时考虑数据链路、任务协同、算法处理、参数保存和通信接口。

实习项目选择 STM32F103 平台作为主要实现平台，主要是因为该平台资料完善、调试环境成熟，能够满足测量数据处理、外设通信和任务调度等需求。本报告中的 STM32 版本定位为工程验证版本，重点完成 SPI 接收、特征包解析、时间差和流量计算、Modbus 应用通信、EEPROM 参数保存以及按键输入等功能的设计与实现。

本项目的软件实现采用 C 语言和模块化设计方法，结合 FreeRTOS 实时操作系统构建多任务运行环境。数据接收、算法处理、Modbus 解析、EEPROM 读写和按键扫描等功能以任务或模块形式组织，并通过信号量、队列和共享参数结构进行协同。通过这样的结构，系统能够将中断响应、DMA 传输、算法计算和通信处理拆分到不同层次中，提高了程序结构的清晰度和运行稳定性。

### 1.1.2 实习单位与实习指导老师

本人实习的单位是杭州睿视智能设备有限公司。公司相关业务与智能设备、嵌入式系统和工业应用场景联系较为紧密，实习期间本人主要围绕嵌入式测量系统中的 MCU 数据处理模块开展学习、开发和调试工作。

实习过程中，企业指导老师为朱赵君老师。指导老师在项目背景、开发流程、嵌入式系统调试方法以及报告撰写思路等方面给予了帮助。学校指导教师赵冬冬老师对实习任务安排、阶段总结和文档规范提出了指导意见，使本人能够将实习内容与计算机科学与技术专业方向形成较好的衔接。

# 第二章 实习任务与完成情况

## 2.1 本人承担的主要工作

在本次实习中，本人承担的工作主要围绕 STM32 数据处理与通信模块展开，具体包括以下几个方面。

第一，完成实习项目的需求分析和功能定位。本人根据项目总体方案，分析 STM32 数据处理模块需要完成的数据接收、算法计算、参数管理和通信输出功能，明确测量数据包进入 MCU 后的处理流程。

第二，负责 SPI 特征数据包接收流程的设计。测量数据源输出的特征包长度为 22 Byte，包含两个相关峰粗定位索引和三点峰值邻域数据。本人设计了基于外部中断、二值信号量、SPI DMA 和 FreeRTOS 队列的接收流程，使 MCU 能够以完整数据包为单位读取测量结果，并将接收任务和算法任务解耦。

第三，负责特征包解析和算法计算模块的整理。本人根据数据包格式恢复 `idxA`、`idxB`、`y1`、`y2`、`y3` 等字段，利用粗定位索引和三点抛物线插值思路估算传播时间差，再结合管道直径、壁厚、安装角度、声速修正等参数计算流速、瞬时流量和累计流量。算法模块还加入了异常范围判断、结果平滑、零漂补偿、小流量切除和报警标志等处理思路。

第四，负责 FreeRTOS 多任务框架的组织。系统中数据接收、算法处理、串口通信、Modbus 解析、EEPROM 读写和按键处理等逻辑并不适合全部写在主循环中。因此，本人将不同功能拆分为多个任务或模块，并通过信号量、队列和共享数据结构传递事件和结果，使程序结构更符合嵌入式实时系统的开发习惯。

第五，参与 Modbus 应用通信框架设计。面向实际现场应用，测量设备通常需要被上位机读取和配置。本人在 STM32 工程中加入了 Modbus 相关中间件和任务框架，采用 USART DMA 环形接收、IDLE 空闲中断判帧、环形缓冲和解析状态机等思路，将流速、瞬时流量、累计流量、报警状态和配置参数等数据组织为便于上位机访问的通信对象。

第六，完成 EEPROM 参数保存框架的设计。为了满足设备参数掉电不丢失的需求，本人在工程中加入 AT24Cxx EEPROM 支持，通过 I2C 驱动、板级封装和独立 EEPROM 任务组织读写流程，并将读写请求封装为异步事件，避免参数保存过程直接阻塞算法任务或通信任务。

第七，加入按键读取模块。实习代码中增加了基于外部中断和 FreeRTOS 队列的按键处理框架，中断层只记录按键边沿事件，任务层负责去抖、短按和长按识别。该模块为本地参数设置和设备调试预留了输入接口。

第八，完成阶段性测试、代码分析和报告材料整理。本人在开发过程中对各模块执行链路进行梳理，记录当前 STM32 版本中已经实现的功能、测试现象和调试过程，并将这些内容整理为实习报告撰写材料。

## 2.2 完成实习任务的技术方案与步骤

### 2.2.1 相关技术

本项目属于嵌入式测量系统软件开发，涉及微控制器平台、实时操作系统、外设通信、数据处理算法和工业通信协议等内容。

STM32F103 微控制器是本次实习项目使用的主要控制平台。该芯片基于 ARM Cortex-M3 内核，集成 GPIO、SPI、USART、I2C、DMA、定时器和外部中断等常用外设，适合用于中小型嵌入式控制和数据处理任务。对于本项目而言，STM32F103 负责组织数据接收、任务调度、算法计算、参数保存和通信输出等核心功能。

FreeRTOS 是本项目采用的实时操作系统。与传统单一主循环相比，FreeRTOS 可以将不同功能拆分为多个任务，使数据接收、算法计算、通信解析和参数保存等操作具有较清晰的执行边界。本项目中使用信号量完成中断与任务之间的同步，使用队列在任务之间传递数据包和操作事件，使用独立任务处理耗时较长或需要等待外设完成的操作。

SPI 是 MCU 接收测量特征数据的重要接口。在本项目设计中，MCU 作为 SPI 主机读取固定长度的 22 Byte 特征包；当测量数据准备完成后，数据就绪信号通过外部中断通知 MCU 读取。DMA 的使用可以减少 CPU 在数据搬运上的占用，使 MCU 更适合同时承担算法和通信任务。

超声波时差式流量测量是本项目的算法背景。该方法通过比较超声波顺流传播和逆流传播时间差，推算流体速度，再根据管道截面积计算流量。本项目中，STM32 接收已经整理好的测量特征数据，并利用这些数据完成时间差细化和工程量换算。

Modbus RTU 是工业现场较常见的串行通信协议。通过将设备内部状态映射为线圈、离散输入、保持寄存器和输入寄存器，上位机可以用统一方式读取测量结果或写入配置参数。本项目的 STM32 版本完成了 Modbus 应用通信框架设计，使测量结果和配置参数能够以更贴近现场应用的方式组织。

EEPROM 参数保存和按键输入属于工程化辅助功能。测量设备中的管道参数、通信地址、补偿参数和报警阈值等配置需要在断电后保持，因此需要非易失存储器。按键输入则为本地菜单、参数调整和功能触发提供可能。虽然这两类功能不直接参与时间差计算，但它们能提高系统的可配置性和现场使用便利性。

### 2.2.2 系统需求分析

从功能需求看，STM32 数据处理模块首先需要完成特征数据接收。测量数据源输出由 `idxA`、`idxB`、`y1`、`y2`、`y3` 组成的定长特征包。MCU 需要保证每次读取的数据完整、顺序正确，并在读取后及时交给算法任务处理。

其次，系统需要完成流体参数计算。接收到的特征包本身并不是用户可以直接理解的工程量，它只包含相关峰位置和局部峰值信息。MCU 需要先根据索引值和采样周期计算粗略传播时间，再利用三点抛物线插值获得亚采样级修正，最终得到传播时间差 `dt`。在此基础上，再结合管道结构参数和安装参数换算流速、瞬时流量和累计流量。

第三，系统需要具备一定的输出稳定性。实际测量中，噪声、异常峰值、安装误差和流体扰动都会影响瞬时计算结果。如果算法直接输出每一帧原始结果，容易导致上位机显示和控制判断出现较大波动。==因此，MCU 端需要加入异常剔除、滑动窗口、质量评价、滤波和限幅等处理方法，使输出结果更平稳。==

第四，系统需要面向实际场景提供通信接口。单纯通过调试串口打印结果只能满足实验观察，不能满足设备部署后的上位机访问需求。因此，实习项目中引入 Modbus RTU 通信思路，将测量结果、报警状态和参数配置组织为标准化寄存器对象，便于上位机或工控系统读取。

第五，系统需要考虑参数保存和本地交互。管道内径、壁厚、安装角、通信地址、滤波参数和报警阈值等配置不应每次上电都重新输入。EEPROM 模块用于保存这些配置参数，按键模块用于本地菜单或调试操作。它们使系统具备更完整的设备软件形态。

### 2.2.3 系统概要设计

根据上述需求，本项目以 STM32 为主控进行软件设计。系统接收固定格式的测量特征数据包，并在 MCU 侧完成数据解析、时间差计算、流量换算、通信输出和参数管理。软件结构可划分为外设驱动层、任务层、中间件层和应用算法层。

外设驱动层主要包括 SPI、USART、I2C、EXTI 和延时等基础驱动。SPI 用于读取测量特征包，USART 用于串口调试和 Modbus 通信，I2C 用于访问 AT24Cxx EEPROM，EXTI 用于接收数据就绪信号和按键边沿事件。

任务层主要包括 SPI 接收任务、算法处理任务、Modbus 解析任务、EEPROM 任务、串口任务和按键任务。各任务之间通过 FreeRTOS 的同步机制进行协作，避免不同功能直接互相调用造成耦合。

中间件层主要包括 Modbus RTU 协议解析、CRC 校验、环形缓冲、EEPROM 设备封装和按键事件封装等内容。这一层把较通用的功能从应用逻辑中分离出来，使上层更关注测量业务本身。

应用算法层主要包括参数结构、特征包解析、时间差计算、流速流量换算、结果滤波、累计流量和报警状态管理。该层是 STM32 数据处理模块的核心，直接决定从测量特征数据到最终测量结果的转换过程。

系统核心数据流可概括如下：

```text
DATA_READY 外部中断
-> 释放数据就绪信号量
-> task_spi_rx 被唤醒
-> 拉低 SPI 片选并启动 SPI DMA 接收
-> 接收 22 Byte 特征包
-> DMA 完成回调释放 xSem_SPI_Rx_Done
-> 封装为 rufx_raw_packet_t
-> 写入 xQueue_Rx_Index_Buf
-> task_algorithm 读取数据包
-> 解析 idxA、idxB、y1、y2、y3
-> 计算 t1、t2 和 dt
-> 换算流速、瞬时流量和累计流量
-> 进行滤波、报警和结果输出
-> Modbus、串口或显示模块读取结果
-> EEPROM 保存配置参数，按键模块预留本地输入
```

这一流程体现了本项目的主要设计思想：中断只负责快速通知，DMA 负责数据搬运，任务负责业务处理，队列负责模块解耦，算法层负责工程量转换。

### 2.2.4 系统详细设计

在 STM32 工程中，实习代码主要位于 `fieldwork` 工程目录下。工程可以按照职责划分为几个部分：`User` 目录负责系统入口和中断入口，`Perph` 目录负责底层外设驱动，`Task` 目录负责 FreeRTOS 任务入口，`App` 目录负责参数结构和算法处理，`Middlewares` 目录负责 Modbus、环形缓冲等中间件，`BSP` 目录负责 EEPROM、按键和串口缓冲等板级支持。这样的结构能够表达出嵌入式软件分层设计的基本思路。

SPI 接收模块是系统数据链路的起点。当一组测量特征数据准备完成后，数据就绪引脚触发 MCU 外部中断。中断服务函数不直接读取 SPI 数据，而是释放一个二值信号量。SPI 接收任务平时阻塞等待该信号量，当信号到来后拉低片选信号，启动 `HAL_SPI_Receive_DMA` 接收固定长度数据包。DMA 接收完成后通过回调释放接收完成信号量，任务再将缓冲区中的字节数据封装为原始特征包，并发送到算法任务队列。这样的设计避免了在中断中执行耗时操作，也减少了主循环轮询数据状态带来的资源浪费。

特征包解析模块负责将字节流恢复为算法可使用的数据结构。根据接口定义，特征包长度为 22 Byte，其中 `idxA` 和 `idxB` 各占 2 Byte，用于表示两帧相关峰的大致位置；`y1`、`y2`、`y3` 各占 6 Byte，用于表示峰值邻域三点相关结果。与直接传输完整波形相比，这种特征包结构明显减少了数据量，使 MCU 只处理与时间差解算直接相关的特征。解析模块按照固定偏移读取字段，并将无符号索引和带符号相关值恢复为内部算法变量。

时间差计算模块建立在“粗定位索引 + 三点插值”的基础上。`idxA` 和 `idxB` 对应两个传播方向或两帧测量中相关峰的整数采样点位置，结合采样频率可以得到基础传播时间。由于真实峰值位置可能落在两个采样点之间，系统进一步使用 `y1`、`y2`、`y3` 进行抛物线插值，估算峰值附近的亚采样偏移。通过该方式，MCU 可以在固定数据包长度下提高时间差估计精度。

流量计算模块负责将传播时间差转换为工程量。算法根据超声波时差法原理，结合管道内径、壁厚、安装角、声速和固定延时等参数，将时间差换算为流体轴向速度，再根据管道截面积计算瞬时体积流量。随后，系统根据采样周期或算法更新周期累加得到累计流量。相关参数采用默认值或配置结构体保存，并可通过通信或本地输入方式进行调整。

输出稳定化模块用于提高结果可用性。实际测量数据常受到噪声和异常峰影响，因此算法处理过程中加入了质量评价、异常范围判断、滑动窗口、IQR 离群值剔除、Kalman 滤波、零漂补偿、小流量切除和上限限幅等思路。实习报告中不需要展开复杂公式推导，但需要说明这些处理的工程意义：它们并不是改变测量原理，而是在结果输出阶段降低偶发异常对显示和通信结果的影响。

Modbus 通信模块主要面向上位机读取和配置需求。STM32 工程中，串口接收采用 USART DMA 环形接收方式，并使用 IDLE 空闲中断判断一帧数据可能已经到达。底层接收缓冲和协议解析任务分离，解析任务从环形缓冲中读取字节，按照地址、功能码、数据区和 CRC 的顺序进行状态机处理。该设计使测量结果、报警状态和配置参数能够以较清晰的协议对象形式组织。

EEPROM 参数保存模块面向设备配置持久化需求。系统使用 AT24Cxx EEPROM 作为非易失存储器，通过 I2C 完成底层访问。为了避免写入 EEPROM 时阻塞算法任务，工程中设计了独立 EEPROM 任务，上层调用异步读写接口后，将读写请求投递到事件队列中，由 EEPROM 任务统一执行。该模块可用于保存管道参数、通信地址、补偿系数、报警阈值和用户配置。

按键输入模块是本地交互功能的组成部分。按键对应的 GPIO 被配置为外部中断输入，中断层只记录按键编号、触发边沿和时间戳，并将事件送入内部队列。按键任务在任务上下文中完成去抖、按下释放顺序判断、短按和长按识别，最后向上层输出统一键值。该结构为本地设置界面和现场调试操作提供了基础。

在工程实现方面，STM32 平台承担了系统主要控制与验证任务。项目重点在于确认软件链路、模块边界和任务协同关系，包括外设初始化、数据包接收、Modbus 寄存器组织、RS485 应用适配思路和实际演示流程等内容。通过这一实现过程，系统形成了较完整的 MCU 软件结构。

## 2.3 实习中的问题及解决方法

### 2.3.1 STM32 工程结构需要兼顾功能实现与维护性

本项目使用 STM32F103 平台实现管道流体测量数据处理功能。由于系统同时包含 SPI 接收、算法计算、Modbus 通信、EEPROM 参数保存和按键输入等模块，如果所有逻辑都集中在主函数或少数文件中，程序会比较难维护，也不利于调试。

针对这一问题，本人在实习中采用模块化组织方式，将外设驱动、任务入口、算法处理和中间件逻辑尽量分开。SPI、USART、I2C、EXTI 等内容集中在外设驱动层，算法计算和参数结构放在应用层，Modbus、环形缓冲和 EEPROM 封装放在中间件或板级支持模块中。通过这种方式，工程结构更加清晰，调试时也更容易定位问题。

### 2.3.2 测量数据到达具有事件性，直接轮询效率较低

测量特征包并不是随时可读，而是在一组测量数据准备完成后才通知 MCU。如果 MCU 采用主循环轮询方式不断检查数据状态，会浪费 CPU 时间，并且容易造成接收时机不清晰的问题。

解决方法是采用外部中断、信号量和 DMA 相结合的事件驱动流程。当测量数据准备完成后，通过中断引脚通知 MCU；中断服务函数只释放信号量，不做复杂处理；SPI 接收任务被唤醒后再启动 DMA 读取完整数据包。这样既保证了响应及时性，又避免了中断函数过长。接收完成后使用队列把数据包交给算法任务，进一步隔离了数据接收和算法处理。

### 2.3.3 测量结果容易受噪声和异常点影响

超声波流量测量中，回波信号质量、相关峰位置、管道安装条件和现场噪声都会影响时间差估计。如果直接把每帧计算结果作为最终输出，瞬时流量可能出现跳变，累计流量也可能受到异常值影响。

针对该问题，本人在算法处理中加入了多级稳定化思路。首先对传播时间差和流量结果设置合理范围，剔除明显不可信的输入；其次引入质量评价和滑动窗口统计，减少单帧异常对结果的影响；再次结合 IQR 离群值剔除和 Kalman 滤波等方法平滑输出；最后对零点附近引入零漂补偿和小流量切除，避免静止状态下出现明显累计误差。通过这些处理，系统输出结果的稳定性得到了改善。

### 2.3.4 Modbus 协议与业务参数映射较复杂

Modbus RTU 不只是串口收发字节，还涉及帧边界判断、CRC 校验、功能码处理、寄存器地址映射、异常响应和与内部参数系统的联动。若将这些逻辑全部写在一个串口回调函数中，代码会很难维护，也不利于功能扩展。

针对这一问题，实习阶段将 Modbus 设计拆分为数据获得和协议解析两层。底层使用 USART DMA 和 IDLE 中断接收字节流，环形缓冲负责保存数据；协议解析任务再按状态机处理地址、功能码、数据区和 CRC。业务参数映射由寄存器表统一关联测量结果、配置参数和动作命令。这样的设计使 STM32 工程形成了较清晰的协议处理框架。

### 2.3.5 参数保存操作可能阻塞实时任务

EEPROM 写入速度相对较慢，并且页写和写周期等待都会占用时间。如果在算法任务或通信任务中直接执行 EEPROM 写入，可能影响系统实时性，尤其是在外部通信请求触发参数保存时更容易造成任务阻塞。

解决方法是设计异步 EEPROM 任务。上层只负责创建读写请求并投递到队列，真正的 I2C 访问由 EEPROM 任务在任务上下文中执行。这样算法任务和通信任务只需要发起请求，不需要等待完整写入过程结束。该设计也便于加入写入完成回调、错误重试和参数版本校验等机制。

第三，算法参数还需要结合实验数据进一步标定。当前流量换算、滤波参数、零漂补偿和异常阈值已经能够支撑流程运行，之后可结合静态测试和不同流速条件下的实验数据进行调整，从而提高结果可信度。

第四，Modbus 功能仍有优化空间。系统已经形成了通信框架和解析流程，可进一步细化输入寄存器、保持寄存器、线圈和离散输入的地址分配，将流速、瞬时流量、累计流量、报警状态、管道参数和动作命令组织成更完整的映射表，并加强上位机测试。

第五，EEPROM 和按键模块还可以与参数系统和本地界面进一步联动。EEPROM 模块可加入参数版本、校验和默认值恢复等机制，按键模块则可与显示界面或菜单逻辑结合，实现本地查看、修改和保存参数。

针对以上问题，后面的优化工作可以从三个方向继续推进：一是围绕实际测量数据完善算法和测试，二是细化 Modbus、EEPROM、按键和显示等工程化功能，三是继续优化代码结构和模块接口。通过这些工作，实习阶段形成的 STM32 工程可以进一步提升稳定性和可维护性。
