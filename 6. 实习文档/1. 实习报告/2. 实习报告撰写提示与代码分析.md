# 实习报告撰写提示与代码分析

## 1. 实习项目定位

本次实习项目可以定位为：**基于 STM32 验证平台的管道流体测量 MCU 数据处理与通信输出模块设计**。

更自然的表述是：该实习项目来源于毕业设计中的 MCU 子系统，将完整毕设系统中的后级处理部分单独抽出，在 STM32 平台上进行前期验证，重点完成前端特征数据接收、时间差解算、流体参数计算、Modbus 场景化输出框架、参数掉电保存和按键输入框架设计。后续再迁移到 GD32F303RCT6 平台，并在演示场景中使用 GD32F303CCT6 板卡进行功能展示。

报告中建议把 STM32 版本写成“原型验证阶段”，不要写成最终量产版本。这样既符合代码状态，也便于解释其尚未重构、部分模块仍为框架或验证用途的情况。

## 2. 与毕业设计的关系

毕业设计完整系统可以概括为“FPGA 前处理 + MCU 后处理”的超声波时差式管道流量测量系统。

FPGA 侧主要负责激励控制、采样同步、卷积或相关运算、峰值搜索和关键特征提取。MCU 侧主要负责读取 FPGA 输出的特征包，完成数据解析、传播时间差求取、流速流量换算、结果滤波、累计流量管理、报警判断以及通信输出。

实习项目正是从完整毕设中抽出的 MCU 后级处理部分，重点不在 FPGA 的波形采样与相关计算，而在以下三件事：

1. 通过 SPI 接收 FPGA 或前端测量模块输出的特征数据包。
2. 根据特征包中的 `idxA`、`idxB`、`y1`、`y2`、`y3` 计算 `t1`、`t2` 和 `dt`，再换算流速、瞬时流量和累计流量。
3. 通过 Modbus 思路将测量结果、报警状态和部分配置参数组织成更贴近实际应用的外部通信接口。
4. 通过 EEPROM 和按键读取等辅助模块，为参数保存、本地交互和现场设备使用习惯预留工程接口。

写报告时可以强调：实习内容不是凭空另做一个项目，而是对毕业设计系统中 MCU 端功能的阶段性抽取、验证和工程化整理。

## 4. 核心数据流

STM32 版本的核心链路可以概括为：

```text
FPGA_INT 外部中断
-> 释放 xSem_FPGA_INT
-> task_spi_rx 被唤醒
-> 拉低 SPI 片选
-> HAL_SPI_Receive_DMA 接收 22 Byte 特征包
-> DMA 完成回调释放 xSem_SPI_Rx_Done
-> 封装为 rufx_raw_packet_t
-> 写入 xQueue_Rx_Index_Buf
-> task_algorithm 读取数据包
-> rufx_unpack_packet 解析字段
-> rufx_calc_t1_t2_dt 计算 t1/t2/dt
-> algorithm_process_group 计算流速、流量、SQ、滤波和报警
-> xQueue_AlgoOut 输出结果
-> Modbus/串口/显示等上层模块读取或展示
-> EEPROM 保存配置参数，按键任务预留本地输入
```

这条链路非常适合写进报告的“系统概要设计”或“系统详细设计”部分。它能体现 FreeRTOS 信号量、队列、DMA、中断和算法任务之间的协同关系。

## 5. SPI 接收模块怎么写

`Task/src/task_spi_rx.c` 是实习项目中最适合重点描述的文件之一。

报告中建议这样写：

> SPI 接收任务采用事件驱动方式运行。当前端 FPGA 完成一帧特征数据准备后，通过 `FPGA_INT` 引脚触发外部中断，中断服务函数释放二值信号量。SPI 接收任务阻塞等待该信号量，当信号到来后拉低片选信号，并启动 SPI DMA 接收固定长度数据包。DMA 接收完成后通过回调释放接收完成信号量，任务随后将接收缓冲区中的字节数据封装为原始特征包，并通过 FreeRTOS 队列发送给算法任务处理。

可以强调的技术点：

- 使用外部中断避免 MCU 周期轮询前端状态。
- 使用 DMA 降低 SPI 接收过程中的 CPU 占用。
- 使用信号量完成中断和任务之间的同步。
- 使用队列隔离数据接收任务和算法处理任务。
- 使用定长特征包，使接收逻辑和解析逻辑边界清晰。

注意：当前代码中 `spi1_dma_init()` 与头文件声明、启动流程接入还不够完整。报告里不要写“完成了稳定量产级 SPI 驱动”，建议写“完成了 SPI 接收任务设计和基于 DMA 的接收流程验证”。

## 6. 算法计算模块怎么写

算法部分主要涉及：

- `App/src/algorithm_packet.c`
- `App/src/algorithm_process.c`
- `App/src/algorithm_flow.c`
- `Task/src/task_algorithm.c`
- `App/src/data.c`

报告中可以按四步写。

第一步，特征包解析。`rufx_unpack_packet()` 将 22 Byte 特征包解析为 `idx_a`、`idx_b`、`conv_y1`、`conv_y2`、`conv_y3`。其中 `idx_a` 和 `idx_b` 用于粗时间定位，三个 `conv_y` 用于峰值邻域插值。

第二步，传播时间差计算。`rufx_calc_t1_t2_dt()` 使用 `65 MHz` 采样频率对应的采样周期，将索引转换为时间，并通过三点抛物线插值估计亚采样偏移，得到传播时间差 `dt`。

第三步，流速与流量换算。`algorithm_process_group()` 调用流量算法，根据管径、壁厚、安装角、固定延时、管壁传播时间等参数，将 `dt` 换算为流速，再由管道截面积计算瞬时流量。

第四步，结果稳定化。算法链路包含异常范围判断、SQ 质量统计、滑动窗口、IQR 离群值剔除、Kalman 滤波、零漂补偿、小流量切除、上限限幅、累计流量更新和报警判断。

这一部分可以和毕业论文第 5 章内容互相呼应。实习报告不需要推导太深，但可以把“从特征包到工程量输出”的链路写完整。

## 7. Modbus 模块怎么写

STM32 实习版本中的 Modbus 相关代码还处于框架阶段，主要包括：

- `Middlewares/Modbus/inc/modbus_protocol.h`
- `Middlewares/Modbus/src/modbus_parse.c`
- `Middlewares/Modbus/src/modbus_crc.c`
- `Middlewares/circular_buffer/src/circular_buffer.c`
- `BSP/USART/src/bsp_usart.c`
- `Task/src/task_modbus_parse.c`

从代码状态看，STM32 版本已经体现了 Modbus 子系统的设计方向，可以拆成“数据获得”和“协议解析”两层来写。

第一层是 Modbus 数据获得。`Perph/src/usart.c` 将 USART1 配置为 `9600-8N1`，使用 DMA 循环接收串口数据，并开启 IDLE 空闲中断。中断中根据 DMA 剩余计数计算当前写指针，再通过 `uart1_idle_irq_callback()` 同步环形缓冲区的 `head` 位置。`BSP/USART/src/bsp_usart.c` 中的串口任务负责创建 `circular_buf_t` 环形缓冲区，并把接收事件进一步通知给 Modbus 解析任务。这个设计体现了“串口接收由 DMA 搬运，中断只负责判定一帧可能到达，协议处理放到任务中完成”的思路。

第二层是 Modbus 协议解析。`Middlewares/Modbus/src/modbus_parse.c` 中的 `task_modbus_parse()` 通过队列等待 UART 任务通知，从环形缓冲区逐字节取出数据，并使用 `modbus_parser_t` 中的状态机字段处理地址、功能码、数据区和 CRC。`modbus_protocol.h` 已经定义了常见功能码、异常码、解析状态和从机地址来源。当前版本中状态机、CRC 更新和功能码处理还没有完全闭环，因此更适合把它写成“通信框架和协议解析思路验证”，而不是写成完整协议栈。

因此报告中建议使用温和准确的表述：

> 在通信应用方面，本人结合工业现场中常见的 Modbus RTU 通信方式，设计了基于 USART DMA 接收、IDLE 中断判帧、环形缓冲区、解析任务和协议状态机的 Modbus 通信框架。该框架将串口数据接收与协议解析进行分离，为后续将流速、瞬时流量、累计流量、报警状态和参数配置映射到 Modbus 寄存器提供了基础。STM32 阶段主要完成协议框架和任务协同验证，后续在 GD32 平台中进一步完善寄存器映射、功能码处理和 RS485 应用适配。

可以借用 GD32 毕设项目中的完善内容来补充“后续完善”或“应用场景适配”：

- 使用 Modbus RTU 作为上位机读取和配置接口。
- 将测量结果映射为输入寄存器。
- 将管径、壁厚、安装角、Modbus 地址等配置映射为保持寄存器。
- 将清零、保存参数、恢复默认参数等动作设计为命令型接口。
- 在 CCT6 演示板上考虑 RS485 方向控制，在 RCT6 目标板上直接使用 UART/RS485 接口适配。

报告中不要写 STM32 版本“已经完整实现 Modbus 多功能码和全部寄存器映射”。更合适的写法是“完成了 Modbus 应用框架设计，并在后续 GD32 版本中进一步完善”。

## 8. EEPROM 参数保存模块怎么写

STM32 实习版本中 EEPROM 相关代码比 Modbus 更接近完整模块，主要涉及：

- `Task/src/task_bsp_e2prom.c`
- `BSP/at24cxx/src/at24cxx_handler.c`
- `BSP/at24cxx/src/at24cxx_driver.c`
- `BSP/at24cxx/inc/at24cxx_handler.h`
- `Perph/src/iic.c`
- `App/src/data.c`

这一部分建议写成“参数持久化与异步存储框架”。`task_bsp_e2prom.c` 创建 EEPROM 任务，并在任务内部初始化 I2C 驱动、AT24Cxx 驱动、设备信息和操作系统适配接口。操作系统适配接口把 FreeRTOS 的延时和队列封装成 EEPROM 模块可使用的函数指针，从而使上层处理逻辑不直接依赖具体 RTOS API。

`at24cxx_driver.c` 负责底层读写，包含参数检查、I2C 资源检查、地址越界判断、页写处理和写周期等待。`at24cxx_handler.c` 则进一步将读写操作封装成事件模型：上层调用 `e2prom_write_async()` 或 `e2prom_read_async()` 后，模块动态创建一个读写事件并投递到事件队列，由 `task_e2prom_handler()` 在任务上下文中真正执行 I2C 访问。

报告中可以这样表述：

> 为满足测量设备对参数掉电保存的需求，系统设计了基于 AT24Cxx EEPROM 的参数持久化模块。该模块通过 I2C 驱动完成 EEPROM 访问，并使用 FreeRTOS 队列将读写请求封装为异步事件，由独立 EEPROM 任务统一处理。这样可以避免参数保存过程直接阻塞算法任务或通信任务，也便于后续将管径、壁厚、安装角、Modbus 地址、报警阈值等配置参数保存到非易失存储器中。

需要注意的写法：

- 可以写“设计并实现了 EEPROM 异步读写框架”。
- 可以写“为参数掉电保存提供了基础”。
- 不要写“已经完成所有参数在实际板卡上的长期可靠性验证”。
- 可以把 `App/src/data.c` 中的 `SaveParameters()`、`LoadParameters()` 与 EEPROM 模块联系起来，说明参数结构体 `g_parameters` 是后续持久化的主要对象。

## 9. 按键读取模块怎么写

按键相关代码主要涉及：

- `Task/src/task_bsp_key.c`
- `BSP/KEY/src/bsp_key.c`
- `BSP/KEY/inc/bsp_key.h`
- `Perph/src/exti.c`

该功能已经加入工程，但当前还没有与显示界面或菜单逻辑形成完整闭环。因此报告里适合写成“本地输入预留模块”或“按键读取框架”，不要写成已经完成成熟菜单交互。

代码中按键模块的设计思路比较清晰：`Perph/src/exti.c` 将 K1-K4 配置为双边沿外部中断，`key_gpio_exti_handler()` 在中断中记录按键编号、触发边沿和时间戳，并把事件指针送入内部队列 `key_inner_queue`。`task_key()` 在任务上下文中接收事件，完成去抖、下降沿/上升沿顺序判断、短按/长按判定，最后把统一的键值送入对外队列 `key_queue`。上层模块可以通过 `key_scan()` 获取最终按键值。

报告中可以这样表述：

> 为后续本地参数设置和菜单操作预留输入接口，系统加入了基于外部中断和 FreeRTOS 队列的按键读取模块。该模块将原始 GPIO 边沿事件与上层按键语义进行分离：中断层只记录按键事件并快速退出，按键任务负责去抖、短按和长按识别，最终向上层输出统一键值。虽然 STM32 阶段该模块尚未与完整菜单界面联动，但其结构已经为后续本地交互功能提供了基础。

可以强调的技术点：

- 中断层轻量化，只记录事件。
- 任务层完成去抖和长短按判断。
- 通过队列隔离中断事件和上层应用。
- `key_scan()` 为后续菜单或显示任务提供统一读取接口。

## 10. 和 GD32 毕设项目的衔接写法

你当时的思路是先用 STM32 做测试，再到 GD32F303RCT6 上完善，演示用 GD32F303CCT6。报告里可以这样叙述：

> 在实习初期，为降低硬件联调复杂度，先选用资料较完整、调试环境成熟的 STM32F103 平台搭建 MCU 后级处理原型，验证 FreeRTOS 任务划分、SPI 数据接收、算法处理、参数保存、按键输入和通信接口组织方式。在原型链路基本明确后，再将相关思路迁移到 GD32F303 系列平台，结合目标板卡资源完善外设初始化、Modbus 通信和实际演示流程。

这样写的好处是：

- STM32 代码初级、不重构的问题变成了“原型验证阶段”的合理现象。
- GD32 版本更完整的 Modbus 映射、RS485 适配、界面菜单和参数管理可以作为“实习成果延伸”或“后续完善方向”使用。
- 答辩时如果被问到为什么报告题目写 STM32 而毕设用 GD32，可以解释为“实习阶段先用 STM32 验证 MCU 子模块，毕设阶段再迁移到 GD32 完整系统”。

## 11. 当前实习报告章节建议

### 11.1 实习项目简介

建议把现有“基于 STM32 的管道流体测量数据处理系统实现”扩展为：

> 本实习项目面向超声波时差式管道流量测量系统中的 MCU 后级处理模块展开。项目以 STM32 微控制器作为前期验证平台，围绕前端 FPGA 输出特征包的接收、解析、时间差解算、流速流量计算、Modbus 通信输出、参数掉电保存和按键输入等功能进行设计与实现。该模块是完整毕业设计系统中 MCU 子系统的阶段性抽取，主要用于验证后级处理链路的可行性，并为后续迁移到 GD32F303 平台提供软件结构和算法流程基础。

### 11.2 本人承担的主要工作

建议写成以下几类：

1. SPI 接收任务设计：完成 FPGA 特征数据包接收流程设计，使用外部中断、片选控制和 DMA 接收实现事件驱动式读取。
2. 特征包解析：根据定长数据包格式解析 `idxA`、`idxB`、`y1`、`y2`、`y3` 等字段。
3. 时间差与流量计算：实现传播时间差计算、流速换算、瞬时流量和累计流量输出。
4. 输出稳定化处理：加入异常判断、SQ 统计、滑动窗口、Kalman 滤波和零漂补偿等处理思路。
5. Modbus 应用适配：设计面向上位机读取与配置的 Modbus 通信框架，包括串口 DMA 接收、环形缓冲和解析任务的组织思路。
6. 参数保存模块设计：设计基于 AT24Cxx EEPROM 的异步读写框架，为测量参数掉电保存提供基础。
7. 按键输入模块设计：加入基于外部中断、队列和任务处理的按键读取框架，为后续本地菜单操作预留接口。
8. 平台迁移准备：对 STM32 原型工程中的任务划分和模块接口进行整理，为 GD32F303 平台完善提供参考。

### 11.3 系统需求分析

可以从五个角度写：

- 实时性需求：前端数据到达后，MCU 需要及时读取并处理，避免丢帧。
- 稳定性需求：流体测量结果容易受噪声和异常峰影响，需要滤波、限幅和异常剔除。
- 应用性需求：实际设备需要被上位机读取和配置，因此需要 Modbus 或类似工业通信接口。
- 参数保持需求：管道参数、补偿参数和通信地址等配置需要掉电后保持。
- 本地交互需求：设备需要预留按键输入能力，用于后续菜单切换、参数调整或功能触发。

### 11.4 系统概要设计

建议放一张流程图或用文字写：

```text
前端 FPGA/测试数据源
-> SPI 接收模块
-> 特征包解析模块
-> 时间差解算模块
-> 流速流量计算模块
-> 结果滤波与报警模块
-> Modbus 通信输出模块
-> EEPROM 参数保存模块
-> 按键输入预留模块
```

- SPI 接收模块

## 12. 可以淡化或转化的问题

STM32 版本是初级原型，代码里有一些不适合在报告中硬写成“完整实现”的地方。建议采用“后续完善”口径。

| 代码现象 | 报告写法建议 |
| --- | --- |
| 工程保留正点原子示例框架痕迹 | 写成“基于成熟示例工程搭建验证平台” |
| SPI 初始化入口和启动流程尚需整理 | 写成“完成 SPI DMA 接收流程设计，后续在目标平台统一外设初始化” |
| Modbus 解析、CRC 和串口任务不完整 | 写成“完成 Modbus 通信框架设计，后续在 GD32 平台完善寄存器映射与功能码处理” |
| EEPROM 读写框架已实现但板级长期验证不足 | 写成“完成参数持久化框架设计，为后续实际板卡验证提供基础” |
| 按键模块已加入但未接入完整菜单 | 写成“完成本地输入预留接口，后续与显示/菜单模块联动” |
| 算法代码仍需重构和单元验证 | 写成“完成核心算法链路的阶段性实现，并形成后续优化方向” |
| STM32 与 GD32 平台不一致 | 写成“STM32 用于前期验证，GD32 用于后续完善和演示” |

答辩时如果被问到代码是否完全闭环，可以回答：

> 实习阶段主要完成的是 MCU 后级处理模块的原型验证和软件链路设计，重点验证 SPI 接收、特征包解析、算法处理、Modbus 应用接口、EEPROM 参数保存和按键输入框架的组织方式。后续毕业设计阶段在 GD32 平台上继续完善硬件适配、协议映射、界面联动和系统联调。

## 15. 参考当前文档时的取舍建议

实习报告重点应该写“我做了什么、怎么组织、解决什么问题、有什么收获”，不要写得像毕业论文那样追求完整理论推导。

建议大量借用的内容：

- FPGA 与 MCU 分工思想。
- 22 Byte 特征包。
- SPI 中断提示 + 主机读取。
- `idxA/idxB/y1/y2/y3 -> t1/t2/dt -> 流速/流量`。
- FreeRTOS 任务、信号量和队列。
- Modbus 用于上位机读取和配置。
- EEPROM 用于参数掉电保存。
- 按键模块用于本地输入预留。

建议少写或不展开的内容：

- FPGA 内部 Verilog 细节。
- LVGL 图形界面细节。
- 完整的论文级数学推导。
- 代码中尚未闭环的细节实现。
- EEPROM 和按键在实际板卡上的长期测试细节。
- “已完成最终产品级系统”这类过满表述。

总体策略：**实习报告写成 MCU 子系统原型验证与应用适配，毕业论文再写完整系统架构和算法原理。**
